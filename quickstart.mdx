---
title: "Quickstart"
description: "Run the full Spring Boot Multitenancy stack locally in minutes and explore tenant-isolated data."
---

<Steps>
  <Step title="Prerequisites">
    Ensure you have the following installed:

    - **Java 21** — required for all three Spring Boot services
    - **Docker** — for running backing services via Docker Compose
    - **Ollama** (optional) — only needed if running chat-service without Testcontainers

    ```bash
    java -version   # should show 21+
    docker --version
    ```

    <Tip>
      If you don't want to install Ollama, you can run chat-service with `./gradlew bootTestRun` instead — it automatically spins up an Ollama container via Testcontainers.
    </Tip>
  </Step>

  <Step title="Clone the repository">
    ```bash
    git clone https://github.com/ThomasVitale/spring-boot-multitenancy.git
    cd spring-boot-multitenancy
    ```
  </Step>

  <Step title="Start backing services">
    From the project root, start all backing services (Keycloak, ChromaDB, Grafana, Loki, Prometheus, Tempo, Grafana Alloy):

    ```bash
    docker-compose up -d
    ```

    This starts:

    | Service | URL | Purpose |
    |---------|-----|---------|
    | Keycloak | http://localhost:8080 | OAuth2/OIDC identity provider |
    | ChromaDB | http://localhost:8000 | Vector store for AI chat |
    | Grafana | http://localhost:3000 | Dashboards (user/password) |
    | Loki | http://localhost:3100 | Log aggregation |
    | Grafana Alloy | http://localhost:12345 | OTLP collector admin |

    <Note>
      PostgreSQL for instrument-service and Ollama for chat-service are managed separately. When you run these services with `bootTestRun`, Testcontainers automatically provisions the required containers.
    </Note>
  </Step>

  <Step title="Configure DNS for tenant subdomains">
    The edge service uses subdomain-based routing. Add these entries to your `/etc/hosts` file (requires `sudo`):

    ```bash
    127.0.0.1       dukes.rock
    127.0.0.1       beans.rock
    ```

    <Warning>
      Without these entries, your browser cannot resolve `dukes.rock` or `beans.rock`, and the edge service routing will not work.
    </Warning>
  </Step>

  <Step title="Run instrument-service">
    Open a terminal in the `instrument-service/` directory and run:

    ```bash
    cd instrument-service
    ./gradlew bootTestRun
    ```

    This starts instrument-service on port **8181** and uses Testcontainers to automatically provision a PostgreSQL 16 database. The service will:
    - Run Flyway migrations on the `default` schema
    - Run per-tenant Flyway migrations on the `DUKES` and `BEANS` schemas
    - Seed demo instrument data for both tenants

    <Note>
      The first run may take longer as Docker pulls the PostgreSQL container image.
    </Note>
  </Step>

  <Step title="Run edge-service">
    Open a second terminal in the `edge-service/` directory:

    ```bash
    cd edge-service
    ./gradlew bootRun
    ```

    This starts the API gateway on port **80**. It handles:
    - Subdomain routing (`dukes.rock` and `beans.rock`)
    - Per-tenant OAuth2 login via Keycloak
    - JWT token relay to downstream services
  </Step>

  <Step title="Run chat-service">
    Open a third terminal in the `chat-service/` directory:

    <Tabs>
      <Tab title="With Testcontainers (recommended)">
        ```bash
        cd chat-service
        ./gradlew bootTestRun
        ```

        Testcontainers automatically starts an Ollama container with the `llama2` model pre-loaded and a ChromaDB container. The service loads tenant documents into ChromaDB on startup.
      </Tab>
      <Tab title="With local Ollama">
        First, install [Ollama](https://ollama.ai/) and pull the llama2 model:

        ```bash
        ollama run llama2
        ```

        Then start the service (ChromaDB is already running via Docker Compose):

        ```bash
        cd chat-service
        ./gradlew bootRun
        ```
      </Tab>
    </Tabs>

    Chat-service runs on port **8282**.
  </Step>

  <Step title="Explore tenant isolation">
    With all three services running, test the tenant isolation:

    **Dukes tenant:**

    Open your browser and navigate to `http://dukes.rock/instruments/`

    You'll be redirected to the Keycloak login page for the `dukes` realm. Log in with:
    - Username: `isabelle`
    - Password: `password`

    You'll see the list of instruments from the Dukes rock band — data from the `DUKES` PostgreSQL schema.

    **Beans tenant:**

    Open a second browser window (or incognito) and navigate to `http://beans.rock/instruments/`

    Log in with:
    - Username: `bjorn`
    - Password: `password`

    You'll see the instruments from the Beans rock band — data from the `BEANS` PostgreSQL schema, completely isolated from the Dukes tenant.

    <Tip>
      Notice that `isabelle` cannot log into `beans.rock` and vice versa — each user belongs to a different Keycloak realm, enforcing tenant isolation at the authentication layer.
    </Tip>
  </Step>
</Steps>
